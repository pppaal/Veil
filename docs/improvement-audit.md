# VEIL Improvement Audit

_Synthesis of 132 findings across 20 parallel audits. Prioritized for a privacy-first, E2E-encrypted, no-recovery messenger aiming to match or beat Signal / SimpleX._

## 1. Executive summary

VEIL has a solid skeleton — clean NestJS module boundaries, a typed error contract, a working Double Ratchet, disappearing messages, a careful device-transfer flow, and a strong-on-paper threat model. But it is not yet best-in-class on the three axes that matter most for a secure messenger: **cryptographic correctness, horizontal-scale reliability, and metadata resistance**. The single most serious defect is a pre-authentication Double Ratchet state mutation that lets one spoofed packet permanently wedge a conversation; alongside it, the realtime gateway holds connection state in process-local maps with no Redis adapter, so the product cannot run more than one API node without silently dropping message delivery. On metadata, the server persists exactly the timing/social-graph/attachment-fingerprint data the threat model claims it doesn't keep, and the rate-limiter is fully bypassable via a spoofed `cf-connecting-ip` header. The good news: a large fraction of the highest-leverage fixes are small (AEAD AAD, the ratchet commit-after-verify reordering, throttler hardening, non-extractable web keys), and the abuse-reporting and key-transparency gaps are well-scoped product work rather than rewrites.

Findings merged across areas where they describe the same root cause.

Recipients silently miss wakes and send latency tracks provider health. | High | M | `apps/api/src/modules/push/*`, `apps/api/src/modules/messages/messages.service.ts` |

## 3. Critical / correctness risks

Real bugs, security holes, or data-loss / privacy leaks.

- **Pre-auth ratchet state mutation (data-loss / DoS, critical).** `decryptMessage` runs `performReceiveDhStep()` and overwrites `rootKey`/`receiveChainKey`/`lastSeenPeerRatchetPub` from the raw 32-byte header before any GCM tag check; `X25519.sharedSecretKey` succeeds for any input so the try/catch never fires. One injected frame permanently wedges the conversation. Fix: try-decrypt-then-commit into scratch state. `apps/mobile/lib/src/core/crypto/lib_crypto_adapter.dart`.
- **Frame header not in AEAD (integrity hole).** `_aesGcm.encrypt/decrypt` are called with no `associatedData`; the routing header and nonce are unauthenticated and malleable. Pass `ratchetPub || counter (|| conversationId || senderDeviceId)` as AAD. `lib_crypto_adapter.dart`, `docs/crypto-envelope-spec.md`.
- **No low-order/all-zero peer-key rejection** before X25519 in `performReceiveDhStep`/`bootstrapSessionFromInbound`. Reject all-zero shared secret and self-equal/zero `incomingPeerPub`. `lib_crypto_adapter.dart`.
- **Rate-limiter fully bypassable** via spoofed `cf-connecting-ip` (checked first, no Cloudflare-range / origin-pull verification). Defeats all `/auth/*`, registration, report, enumeration limits. `cf-throttler.guard.ts`.
- **Realtime gateway is single-node only** — cross-node delivery, presence, push-suppression, and forced-logout silently wrong with >1 replica. `realtime.gateway.ts`, `main.ts`.
- **Sockets ignore token expiry/revocation** after connect; live delivery survives logout and 1h TTL. `realtime.gateway.ts`.
- **WS typing handlers skip membership authorization** — any authed socket can probe arbitrary `conversationId`s, inject typing state, and learn membership; no DTO validation on WS payloads. `realtime.gateway.ts`.
- **Unauthenticated key-bundle/profile endpoints** (`GET /users/:handle`, `/key-bundle`) enable handle namespace + full device-graph/key harvesting; only control is the bypassable throttle. `users.controller.ts`, `users.service.ts`.
- **Server-side key-injection with no transparency** — server returns the device set the client trusts; safety number covers only one identity key, so an injected sibling device never trips a warning. `users.service.ts`, `safety_numbers.dart`.
- **No one-time prekeys / no signed-prekey rotation** — no FS for the X3DH pre-key message; a leaked signed-prekey key is valid forever. `users.service.ts`, `lib_crypto_adapter.dart`, `devices.controller.ts`.
- **Abuse reports are write-only; `UserStatus.locked` never set** — no enforcement loop for CSAM/harassment. `safety.service.ts`, `schema.prisma`, `auth.service.ts`.
- **Blocks only stop direct-message delivery** — presence, typing, group messages, and sync fan-out ignore blocks; `blockerIdsFor()` is dead code. A blocked abuser routes around via shared groups. `safety.service.ts`, `messages.service.ts`, `realtime.gateway.ts`.
- **Metadata retained forever** despite threat-model claims: `serverReceivedAt`, receipts (`readAt`/`deliveredAt`), `CallRecord`, device last-seen, plus attachment `sha256`/size/MIME and `UserContact` social graph. DB compromise yields the full timing + social graph. `schema.prisma`, `conversations.service.ts`, `attachments.service.ts`, `docs/threat-model.md`.
- **Web demo extractable keys + plaintext JWK in IndexedDB + wide-open CSP** — XSS exfiltrates permanent identity keys and decrypted plaintext. `apps/web-demo/app.js`, `index.html`.
- **Decrypt failures return fake plaintext strings** (`'[Decryption failed]'`), making a GCM tamper/forgery indistinguishable from a real message and rendering attacker-influenced text in the timeline. Return a typed failure. `lib_crypto_adapter.dart`.
- **Client/server have no request timeouts** — a stalled server hangs the outbox drain forever (`_isDrainingOutbox` stuck true), and slow Prisma/Redis can pin the Node event loop with no 503 shedding. `veil_api_client.dart`, `pinned_http_client.dart`, `apps/api/src/main.ts`.
- **EphemeralStore hard-fails (not falls back) when configured Redis is down** — refresh-token rotation, device transfer, and WS jti-blacklist throw 500/drop; errors silently swallowed by the no-op handler. A Redis blip becomes a fleet-wide auth outage. `ephemeral-store.service.ts`.
- **Replay protection rewinds on crash** — debounced 300ms snapshot flush means a crash re-opens a replay window for messages decrypted since the last flush; no message-id dedup backstop. `lib_crypto_adapter.dart`.
- **Session controller recreated on cache-epoch change** drops bootstrap + in-flight device-transfer state (pending claim silently lost). `apps/mobile/lib/src/app/app_state.dart`.
- **Bootstrap error traps user on splash permanently** — no retry, relaunch re-runs the same failing path; bricks the app. `splash_screen.dart`, `app_router.dart`.
- **No real-DB tests on PR** — even the "e2e" suite stubs Prisma with a 1049-line fake (`$transaction` is a pass-through), so real atomicity / device-transfer TOCTOU / idempotency is never exercised on PRs. `test/app.e2e-spec.ts`, `test/support/fake-prisma.service.ts`, `.github/workflows/ci.yml`.

## 4. Quick wins (high value, low effort)

- **AEAD AAD on the frame header** — small change, closes most of the ratchet/header DoS surface. `lib_crypto_adapter.dart`.
- **Reject all-zero/low-order X25519 outputs** — constant-time compare after each `sharedSecretKey`. `lib_crypto_adapter.dart`.
- **Harden `CfThrottlerGuard`** — only trust `cf-connecting-ip` from Cloudflare ranges / verified origin-pull; fail closed when `VEIL_TRUST_PROXY` is false. `cf-throttler.guard.ts`.
- **Per-request `.timeout()` on the mobile API client** and a connect timeout on the non-pinned branch — unwedges the outbox. `veil_api_client.dart`, `pinned_http_client.dart`.
- **Splash "Retry" button** calling `bootstrap()` after clearing `errorMessage`. `splash_screen.dart`.
- **Web demo:** non-extractable keys, stop writing `edPrivJwk/xPrivJwk`, tighten CSP to `connect-src 'self'`. `apps/web-demo/app.js`, `index.html`.
- **Add `@@index([expiresAt])` (partial) on `messages`** — the sweep already exists; just give it an index. `schema.prisma`.
- **Push observability** — inject logger/counter into `PushService`; redaction for `pushToken` already exists. `push.service.ts`.
- **Add `apns.headers` (`apns-push-type: background`, `apns-priority: 5`) to the FCM override** so iOS-via-FCM wakes aren't dropped. `fcm-push.provider.ts`.
- **Route raw exceptions through the typed error contract** in `safety.service.ts` / `conversations.service.ts`; run `request.url` through the existing redaction in `ApiExceptionFilter`. `safety.service.ts`, `conversations.service.ts`, `api-exception.filter.ts`.
- **Add `statusCode` to `request.completed` log** + error-level on 4xx/5xx so failures are greppable by `requestId`. `logging.interceptor.ts`.
- **`@CurrentUser()` param decorator** to delete `@Req()` boilerplate across controllers. `common/guards/`.
- **Reaction button/chip `Semantics(button, label)`** + localized AppBar tooltips — core interactions currently unusable with a screen reader. `reaction_picker_widget.dart`, `chat_room_screen.dart`.
- **`VeilBlur` should read the palette from context** — today it hardcodes dark, producing near-black title text on a dark bar in the default light theme. `veil_ui.dart`.
- **Report de-dup unique index** `(reporterUserId, reportedUserId[, messageId])` so the report queue can't be weaponized. `schema.prisma`, `safety.service.ts`.
- **Add full jitter to client retry backoff** to avoid a synchronized thundering herd on relay recovery. `veil_messenger_controller.dart`.
- **Jest fake timers** for the typing-throttle test (kills a 520ms wall-clock flake). `realtime.gateway.spec.ts`.

## 5. Path to best-in-class (vs Signal / SimpleX)

### Bet 1 — Cryptographic correctness & verifiable identity
Signal's edge is a ratchet that is correct under adversarial input plus a usable verification story. VEIL must (a) adopt try-decrypt-then-commit with AAD-authenticated headers and low-order-key rejection (Top-10 #1), (b) add one-time prekeys + rotating signed prekeys with keyId/expiry to get X3DH forward secrecy (#7), and (c) make identity tamper-evident — near-term, have each user's devices sign their authorized device set and verify it client-side with an in-conversation "device set changed" banner; medium-term, a CONIKS/Signal-KT-style append-only transparency log. Also upgrade the safety number to a versioned, iterated, identifier-bound hash (Signal uses ~5200 iterations) instead of a single un-domain-separated SHA-512. Without these, "verify safety numbers" gives users false assurance against the exact server-injection attack the model claims to defend.

### Bet 2 — Reliability at scale
Today the product is single-node by construction: in-memory socket maps (#2), per-replica prune sweeps and dead per-replica metrics, no request timeouts, and a Redis layer that hard-fails instead of degrading. Best-in-class means a Redis-backed Socket.IO adapter with rooms, a single coordinated retention job behind an advisory lock, durable post-commit delivery intents (not fire-after-commit fan-out that's lost on crash), per-request timeouts with 503 shedding on both client and server, and a real-DB integration suite on every PR. This is the difference between a demo and an operable service.

### Bet 3 — Metadata resistance (the SimpleX bar)
SimpleX's whole pitch is "no user identifiers, minimal metadata." VEIL currently undercuts its own privacy claims by persisting precise timing, receipts, call records, device last-seen, attachment plaintext fingerprints, an explicit `UserContact` social graph, and per-user request timing in durable logs — all reconstructable on DB compromise regardless of E2E. The path forward: pad message length into buckets client-side; coarsen/forgo `serverReceivedAt` and receipt timestamps (ordering already comes from `conversationOrder`); hash attachments over ciphertext only and drop precise size/MIME; move contacts to a client-encrypted opaque blob; prune `CallRecord`/device-last-seen on a timer; pseudonymize or short-retain `actorUserId` in logs; and document the push-provider wake-timing side channel honestly. Each item is independently shippable, and together they close the gap between the threat-model prose and the schema.

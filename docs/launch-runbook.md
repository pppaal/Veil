# VEIL Launch Runbook (Korea, public launch)

This runbook is the single reference for taking VEIL from internal build to public store submission. Work top-down. Nothing in the "launch gate" section is optional.

## 0. Snapshot of launch-ready artifacts

- Store copy: [`store/play/`](../store/play/) (title, short, full in KO + EN), [`store/appstore/`](../store/appstore/) (metadata KO/EN + App Privacy questionnaire answers).
- Privacy policies: [`docs/privacy-policy-ko.md`](privacy-policy-ko.md), [`docs/privacy-policy-en.md`](privacy-policy-en.md). Host both at `https://veil.app/privacy` (KO default, EN fallback negotiated via Accept-Language).
- iOS privacy manifest: [`apps/mobile/ios/Runner/PrivacyInfo.xcprivacy`](../apps/mobile/ios/Runner/PrivacyInfo.xcprivacy).
- Launcher icons + splash: generated by `flutter_launcher_icons` and `flutter_native_splash` from `assets/branding/`.
- Korean localization: `lib/src/l10n/app_ko.arb` (~120 keys) → generated `AppLocalizations`.
- Local crash logger: [`apps/mobile/lib/src/core/diagnostics/crash_log_service.dart`](../apps/mobile/lib/src/core/diagnostics/crash_log_service.dart) — disk-only, no network egress.

## 1. Pre-flight build checks

```
cd apps/mobile
puro flutter gen-l10n
puro flutter analyze
puro flutter test
```

All three must finish clean. Any warning in analyze must be resolved, not suppressed.

## 2. Version + build numbers

- Bump `version:` in `apps/mobile/pubspec.yaml` following `MAJOR.MINOR.PATCH+BUILD`.
  - `0.1.0+1` → first public release candidate: `1.0.0+1`.
- Tag the repo: `git tag v1.0.0 && git push --tags` (only after the store artifacts below are cut — tags should correspond 1:1 to store builds).

## 3. Android (Play Store)

1. Generate upload keystore if not done: `keytool -genkeypair -v -keystore veil-upload.jks -keyalg RSA -keysize 4096 -validity 36500 -alias veil`.
2. Store keystore + `key.properties` **out of the repo**. Reference the path from `android/key.properties` locally or from CI secrets.
3. Build: `puro flutter build appbundle --release --flavor prod --dart-define=VEIL_ENV=prod`.
4. Upload `build/app/outputs/bundle/prodRelease/app-prod-release.aab` to Play Console internal track.
5. Store listing → paste from [`store/play/title-ko.txt`](../store/play/title-ko.txt), [`short-ko.txt`](../store/play/short-ko.txt), [`full-ko.txt`](../store/play/full-ko.txt) (and `-en.txt` equivalents) into KO and EN language slots.
6. Data safety form → mirror the declarations in [`store/appstore/app-privacy-answers.md`](../store/appstore/app-privacy-answers.md). Key calls:
   - Data collected: User ID, Device ID (both "App functionality", not tracking).
   - Messages encrypted end-to-end: declared as "User content", "Encrypted in transit" + "Encrypted at rest", "No".
   - No advertising, no third-party analytics.
7. Content rating: "Communication" category, self-assessment: no violence, no user-generated profanity moderation claim.
8. Target audience: 14+ (PIPA alignment).

## 4. iOS (App Store)

1. Bump `MARKETING_VERSION` + `CURRENT_PROJECT_VERSION` in `ios/Runner.xcodeproj/project.pbxproj` to match pubspec.
2. Confirm `PrivacyInfo.xcprivacy` is in the Runner target ("Add to target" must be checked).
3. Build: `puro flutter build ipa --release --dart-define=VEIL_ENV=prod`.
4. Upload via Transporter or Xcode → App Store Connect.
5. App Store Connect → App Privacy → answer each question using [`store/appstore/app-privacy-answers.md`](../store/appstore/app-privacy-answers.md) as the source of truth.
6. Listing content → paste from [`store/appstore/metadata-ko.txt`](../store/appstore/metadata-ko.txt) and `metadata-en.txt`.
7. Export compliance: the app uses standard crypto (X25519, AES-GCM, Ed25519). Tick "Standard encryption algorithms (exempt)" and confirm the annual self-classification report filed with the U.S. Bureau of Industry and Security (BIS) — see `docs/production-deployment.md` for the process if not already filed.
8. Age rating: 12+ (user-generated content, unfiltered by us).

## 5. Backend production deploy

1. Ensure `apps/api/.env.production` is populated (see `docs/production-deployment.md`).
2. Run migrations against prod DB:
   ```
   cd apps/api
   npx prisma migrate deploy
   ```
3. Deploy image (Dockerfile in `apps/api/`) behind TLS at `api.veil.app`.
4. Verify health probe `/healthz` and `/readyz`. Turnstile realtime socket should connect with a valid JWT.

## 6. Launch gate — must be green before Submit for Review

- [ ] `puro flutter analyze` is clean.
- [ ] `puro flutter test` is green (78/78 currently).
- [ ] Privacy policy URL resolves publicly with both KO + EN.
- [ ] In-app privacy consent screen lists every field the server actually stores.
- [ ] Account deletion path in Settings → verified end-to-end against staging DB.
- [ ] Korean UI audited on a real device at `ko-KR` locale (splash, onboarding warning, privacy consent, create account, choose handle, settings, chat, conversation list).
- [ ] No-recovery warning screen is shown on first launch before key material is generated.
- [ ] Device transfer flow tested on two physical devices.
- [ ] Crash log writes to disk on forced error and does not leak identifiers to network (verify with proxy/packet capture).
- [ ] TLS pinning or at minimum strict certificate validation confirmed on iOS and Android.
- [ ] App Store export-compliance self-classification filed.
- [ ] Play Store Data Safety declarations match privacy manifest.

## 7. Post-submission

- Watch review status daily. Respond to reviewer queries within 24h.
- On approval, do a phased rollout: 5% → 20% → 50% → 100% over a week, gated on crash log volume and API error rate.
- If a regression surfaces, halt rollout rather than patch-ship. PIPA requires auditable consent flow changes; a broken consent screen is a reportable incident.

## 8. Remote push integration (FCM + APNs)

The app is wired with `RemotePushService` + `PushTokenCoordinator` but ships with a `DisabledRemotePushService` default. To enable real delivery:

### Server side

1. Set in `apps/api/.env.production`:
   ```
   VEIL_PUSH_PROVIDER=fcm             # or apns
   VEIL_PUSH_DELIVERY_ENABLED=true
   VEIL_FCM_PROJECT_ID=<gcp-project-id>
   VEIL_FCM_SERVICE_ACCOUNT_JSON=<minified service account JSON>
   # or for APNs:
   # VEIL_APNS_TEAM_ID=...
   # VEIL_APNS_KEY_ID=...
   # VEIL_APNS_BUNDLE_ID=app.veil.messenger
   # VEIL_APNS_PRIVATE_KEY=<PEM contents>
   ```
2. The `push` module picks the provider at boot — no code change.

### Mobile side

1. Add `firebase_core: ^3.x` and `firebase_messaging: ^15.x` to `apps/mobile/pubspec.yaml`.
2. Run `flutterfire configure` (Firebase CLI) for both Android (`google-services.json`) and iOS (`GoogleService-Info.plist`). Do **not** commit these files — reference via CI secrets.
3. Create `lib/src/core/notifications/firebase_remote_push_service.dart` implementing `RemotePushService`:
   - `fetchToken()` → `FirebaseMessaging.instance.getToken()`
   - `tokenRefresh` → `FirebaseMessaging.instance.onTokenRefresh`
   - `onMessage` → `FirebaseMessaging.onMessage.map((msg) => RemotePushHint(kind: msg.data['kind'] ?? 'message', conversationId: msg.data['conversationId']))`
4. Override the provider in `main.dart`:
   ```dart
   await Firebase.initializeApp();
   final pushService = FirebaseRemotePushService();
   runApp(ProviderScope(overrides: [
     remotePushServiceProvider.overrideWithValue(pushService),
     crashLogServiceProvider.overrideWithValue(crashLog),
   ], child: const VeilApp()));
   ```
5. iOS: enable the "Push Notifications" and "Background Modes → Remote notifications" capabilities in Xcode. Upload the APNs auth key (.p8) to the Firebase console.
6. Android: add `POST_NOTIFICATIONS` permission (API 33+). Add the Firebase gradle plugin per flutterfire docs.

### Verification

- Log in on a test device → observe `POST /devices/push-token` in server logs.
- Send a message from a second device → device stays backgrounded → notification should arrive (metadata-only payload, no message content).
- Revoke the device via settings → verify `DELETE /devices/push-token` is called.

## 9. Incident playbook (minimum)

- **Privacy breach suspected**: disable affected API endpoint, post a Korean-language notice within 72h per PIPA §34 ③.
- **Crypto bug**: force an in-app notice via the banner surface (remote config), do not roll out a silent fix.
- **Account-deletion failure**: treat as P0 — PIPA §36 grants right to erasure on request.
